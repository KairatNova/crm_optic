# Резервное копирование PostgreSQL

Краткая памятка для продакшена (Railway, Render, VPS и т.д.).

## Ручной дамп (`pg_dump`)

На машине с доступом к БД (или через SSH-туннель):

```bash
pg_dump --format=custom --file=crm_optic_$(date +%Y%m%d).dump "$DATABASE_URL"
```

Восстановление в пустую БД:

```bash
pg_restore --clean --if-exists --no-owner --dbname="$DATABASE_URL_TARGET" crm_optic_YYYYMMDD.dump
```

Перед восстановлением создайте целевую БД и пользователя с нужными правами. После восстановления при необходимости выполните `alembic upgrade head` (если схема на цели старее миграций в репозитории).

## Платформы

- **Railway / Render:** включите встроенные автоматические бэкапы Postgres в панели сервиса БД, задайте окно хранения и периодичность по политике компании.
- **VPS:** планируйте `cron` + `pg_dump` в защищённое хранилище (S3-совместимое облако, отдельный сервер). Дампы не храните только на том же диске, что и инстанс БД.

## Практика

- Делайте тестовое восстановление на отдельную БД хотя бы раз после настройки пайплайна.
- Храните секреты (`DATABASE_URL`) в переменных окружения, не в репозитории.
